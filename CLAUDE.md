# pg-loadgen — Claude Code Context

## What this is

A Go-based PostgreSQL load generator. Hammers a database with a realistic mixed workload (INSERT, READ, JOIN, UPDATE, DELETE) using large JSONB payloads to stress heap I/O, Toast storage, and MVCC. Runs as multiple replicas on Docker Compose or Kubernetes. Each replica exposes a Prometheus `/metrics` endpoint.

## Module structure

```
pg-loadgen/
├── main.go                   — wires everything; signal handling; graceful shutdown
├── config/config.go          — all env-var config, single Config struct
├── db/
│   ├── pool.go               — pgxpool setup (MaxConns = Workers+5)
│   └── schema.go             — advisory-lock migration; CREATE TABLE/INDEX IF NOT EXISTS
├── workload/
│   ├── ring.go               — shared circular buffer of recently inserted session UUIDs
│   ├── payload.go            — two pre-rendered template pools (100 templates each); micro-mutation engine
│   ├── ops.go                — all 5 DB operations (Executor struct)
│   ├── worker.go             — RunWorker goroutine; per-worker *rand.Rand
│   └── stats.go              — per-worker stats; 30s rolling summary printer
└── metrics/
    ├── metrics.go            — OpsTotal (Counter), OpDuration (Histogram), WorkersActive (Gauge)
    ├── pool_collector.go     — custom prometheus.Collector; calls pool.Stat() only at scrape time
    ├── index_stats.go        — table stats (always) + index stats (CREATE_INDEXES=true only)
    └── pg_stats.go           — bgwriter + WAL instance-level stats
```

## Key design decisions

### Payload pools
`workload/payload.go` has **two** template pools, not one:
- `sessionTemplatePool` — 4–8 KB (for `sessions.metadata`)
- `eventTemplatePool` — 8–16 KB (for `events.payload`)

Pass `minKB=4` for sessions, `minKB=8` for events. `GetMutatedPayload` picks the pool based on `minKB <= 4`. The `_pad` field brings templates to target size at init; `trace_id` is mutated (16 hex chars) on every use to bust Postgres Toast deduplication.

`audit_log.diff` is generated by `buildAuditDiff` and padded to 2–4 KB.

### Ring buffer
`workload/ring.go` — INSERT workers call `ring.Push(sessionID)` after commit. UPDATE/DELETE/READ workers call `ring.Sample(rng)` to get a known session UUID. This eliminates `ORDER BY random()` full scans. If the ring is empty (cold start), workers return `nil` and skip the op.

### Metrics pattern
- **Hot-path counters**: `metrics.RecordOp` is called per op in worker.go — cheap Counter/Histogram updates only.
- **Pool stats**: custom `PoolCollector` calls `pool.Stat()` once per Prometheus scrape, never inside workers.
- **Polling loops** (`RunTableStatsLoop`, `RunIndexStatsLoop`, `RunPGStatsLoop`): background goroutines that tick on `INDEX_STATS_INTERVAL_SECS`. All delta-track cumulative Postgres counters using `indexScanTracker` / `pgStatsTracker` so Prometheus `rate()` works.
- **30s summary**: `workload/stats.go` — per-worker struct with no mutex on the hot path; brief lock only during `snapshot()`. Merges fixed-bucket histograms (same 10 bounds as Prometheus) to compute p50/p95/p99.

### Advisory lock migration
`db/schema.go` — exactly one pod runs DDL. Others poll `pg_tables` with 500ms sleep until all three tables exist. Lock ID is `7654321` (hardcoded constant).

### Transaction design
- **INSERT**: BEGIN → sessions → 1–3 events → audit_log → COMMIT → ring.Push
- **UPDATE**: `SELECT id FOR UPDATE SKIP LOCKED` (no deadlock possible) → UPDATE sessions (JSONB rewrite forces Toast churn) → audit_log → COMMIT
- **DELETE**: ring-sampled session → `DELETE FROM events WHERE id IN (SELECT ... LIMIT batch_size)` — no time interval arithmetic
- **No `ORDER BY random()` anywhere**

```bash

docker compose down && rm -rf ./pgdata && docker compose up --build
```

## Important constraints

- **Op percentages must sum to 100**: `WRITE_PCT + READ_SIMPLE_PCT + READ_JOIN_PCT + UPDATE_PCT + DELETE_PCT = 100`. Config loading returns an error otherwise.
- **No GIN indexes**: intentional. GIN on 8–16 KB JSONB generates thousands of WAL entries per INSERT, turning the bottleneck into index maintenance rather than the workload under test.
- **PG14+ required** for `pg_stat_wal` (used by `RunPGStatsLoop`). Docker Compose uses postgres:16.
- **`pg_stat_bgwriter` schema changed in PG17**: `checkpoints_timed/req` and buffer checkpoint columns moved to `pg_stat_checkpointer`. The current `collectBgwriterStats` query targets PG14–16.
- **`pgdata/` is gitignored and dockerignored**: it's a local bind mount for the Compose Postgres data directory.
- **Vendor directory is committed**: dependencies are in `vendor/`. Run `go mod vendor` after adding new deps.

## Adding a new metric

1. Declare the `prometheus.*` var in the appropriate `metrics/` file.
2. Register it in the relevant `Register*()` function called from `main.go`.
3. If it's a delta-tracked counter from a Postgres cumulative stat, add a key to the relevant tracker's `lastSeen` map in the collection function.
4. If it needs a new polling loop, follow the `RunTableStatsLoop` / `RunPGStatsLoop` pattern: create tracker inside the loop function, tick on context, log errors only when `ctx.Err() == nil`.

## Adding a new workload op

1. Define a constant in `workload/ops.go` (`OpFoo = "foo"`).
2. Add `OpFoo` to `allOps` slice in `workload/stats.go`.
3. Add a `pct` field to `Config` and wire it into `SelectOp` in `workload/worker.go`.
4. Add `getEnvInt("FOO_PCT", N)` in `config/config.go` and update the sum validation.
5. Implement `(e *Executor) doFoo(ctx)` in `ops.go`.
